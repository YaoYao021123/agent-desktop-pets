系统流程

Codex local data读取本机 Codex quota 与 activity，不发到外部服务。
Python bridge生成小 JSON 包，包含 state、tokens、primary、secondary。
macOS app host通过本地 .app wrapper 获取蓝牙权限并自动重试。
BLE NUS向 StickS3 的 Nordic UART Service 写入状态包。
StickS3 UI显示 David pet、5h/7d 额度条、重置倒计时和状态。

过程记录

1. 确认硬件连接
设备通过 USB-C 被 macOS 识别为 /dev/cu.usbmodem* 串口，同时通过 Codex-XXXX BLE 名称广播。
2. 下载与本地化依赖
PlatformIO 下载过慢，因此将 ESP32 Arduino framework、工具链、文件系统工具和 Arduino libraries 本地安装或 vendored 到项目内。
3. 转换 Codex pets
新增 tools/convert_codex_pet.py，把 ~/.codex/pets 的 atlas 转为 StickS3 需要的 GIF + manifest 结构。
4. 选择 David 作为默认 pet
将 characters/david 打包进 LittleFS，并在 build flags 中设置 DEFAULT_CHARACTER="david"。
5. 编译与刷机
固件编译成功，直接使用 esptool.py --before no_reset 写入 firmware 和 LittleFS，两个写入都完成 hash 校验。
6. 修复 macOS BLE
新增本地 CodexUsageBridgePython.app 和 runner，解决 Python 直接访问 CoreBluetooth 被 macOS TCC 终止的问题。
7. 稳定 BLE 连接
由于 passkey pairing 在 Python CoreBluetooth 下会卡住，当前固件改为开放 NUS 写入，只发送本地使用状态，硬件审批暂不启用。

技术取舍

当前工作版本为了稳定显示额度，关闭了 BLE passkey pairing 和硬件审批代理。传输内容限于本机 Codex 使用百分比、token 数和状态字符串；不连接外部服务器。后续如果要做 A/B 硬件审批，可以在单独分支恢复加密配对或改走 USB serial。
