Agent Desktop Pets / StickS3 communication

通信不是只能靠线，正式运行推荐 BLE。

USB-C 负责刷机、上传 LittleFS、供电和串口调试；Mac 到 StickS3 的 Codex 状态同步，在当前项目里主要通过 BLE Nordic UART Service 完成。

结论 插线不是运行时数据通信的硬要求。刷完固件后，只要 StickS3 有电并且 Mac 蓝牙可用，就可以通过 BLE 收到 Codex quota 和 pet state。USB-C 仍然建议接着，因为 StickS3 电池小，长时间桌面显示更稳。

当前链路怎么分工

M5Stack 官方页面把 StickS3 定义为 ESP32-S3-PICO-1-N8R8 设备，并列出 2.4GHz Wi-Fi 能力；Espressif 的芯片资料还说明这颗 SiP 支持 Bluetooth LE。也就是说，硬件层面并不只有线。当前仓库已经落地的是 BLE 和 USB Serial，Wi-Fi 目前只是硬件能力，还没有接成 Codex 数据通道。

推荐运行链路 BLE 数据同步

Codex 插件启动本地 bridge，扫描名为 Codex-XXXX 的 StickS3，并向 BLE NUS RX characteristic 写入小 JSON 包。

必须用于刷机 USB-C 线

用于 PlatformIO 上传固件和 LittleFS，也可以作为供电线。运行时不一定需要用线传数据。

调试备用 USB Serial

固件会读 Serial 上的 JSON 行，所以可以写测试脚本验证屏幕更新。但当前自动 Codex bridge 默认不是串口版。

硬件能力 vs 当前实现

可以理解成三层：USB-C 是刷机和稳定供电；BLE 是当前项目已经跑通的 Mac 到 StickS3 状态同步；Wi-Fi 设备支持，但本仓库暂时没有写 Wi-Fi server/client，也没有 Codex Wi-Fi bridge。

场景 是否需要 USB-C 是否需要 BLE 说明
刷固件 需要，且要数据线 不需要 pio run -e m5stack-sticks3 -t upload
上传 pet GIF / LittleFS 推荐 USB-C 不需要 pio run -e m5stack-sticks3 -t uploadfs 或 tools/flash_character.py
显示 Codex quota / state 只需要供电 需要 当前稳定方案：Mac bridge 每隔几秒通过 BLE 发包。
快速验证屏幕能否收包 可用 USB Serial 可不用 固件同时监听 Serial 和 BLE；串口适合排错。
完全无线桌面显示 不需要数据线，但需要电池供电 需要 可以无线跑，但 StickS3 电池只有约 250mAh，建议长时间使用时接 USB 电源。

推荐运行方式：USB 供电 + BLE 数据

1. 插 USB-C主要为了供电，也便于后续刷机和看串口日志。
2. 固件启动StickS3 开机后会显示 Codex usage monitor，并广播 Codex-XXXX。
3. Mac 开蓝牙macOS Bluetooth 必须打开，Python bridge 需要 bleak。
4. Codex hook发送任意 Codex prompt，插件 hook 会启动后台 bridge。
5. 屏幕更新StickS3 收到 JSON 后更新 pet state、5h/7d quota 和 token 数。

手动启动 bridge

如果不想等 Codex hook，或者正在排错，可以在仓库根目录手动启动：

python3 plugins/codex-usage-stick/scripts/start_bridge.py --status
python3 plugins/codex-usage-stick/scripts/start_bridge.py --foreground

成功时，日志里会出现类似：

sent {"state":"busy","tokens":...,"primary":...,"secondary":...}

查看日志：

tail -n 20 ~/.codex/codex-usage-bridge/hook.log
tail -n 40 ~/.codex/codex-usage-bridge/bridge.log

USB Serial 是什么角色

固件并没有把 USB 串口废掉。代码里 Serial.begin(115200)，并且 dataPoll() 同时读取 Serial 和 BLE ring buffer。也就是说，StickS3 可以从 USB 串口收 JSON 行。

适合做什么

  • 验证设备是否被 Mac 识别。
  • 验证屏幕是否能响应 JSON 状态包。
  • 调试 character 文件传输和 ACK。
  • 未来实现一个串口版 Codex bridge。

现在不适合做什么

  • 当前 Codex 插件自动启动的不是串口 bridge。
  • 不能只靠插线就自动显示 quota，除非你运行串口发送脚本或后续补一个 serial bridge。
  • 串口设备名在 macOS 上可能是 /dev/cu.usbmodem*，旧测试脚本只扫 /dev/cu.usbserial-* 时可能需要改一下。

串口快速验证

先确认端口：

ls /dev/cu.usbmodem* /dev/cu.usbserial* 2>/dev/null

如果要手动发一个 Codex 状态包，可以用串口工具或临时脚本写入一行 JSON：

{"state":"busy","tokens":159297887,"primary":62,"secondary":44,"now":1778671200}

BLE 连接排错

  1. 确认 StickS3 已经刷入本项目固件，开机后会广播 Codex-XXXX。
  2. 确认 Mac 蓝牙已开启，且安装了 bleak：python3 -m pip install bleak。
  3. 运行 start_bridge.py --foreground，看能否扫描并连接。
  4. 如果 macOS 缓存了旧 BLE 名称，在 ~/.codex/codex-usage-bridge/config.json 里设置 address，或删除旧配对记录。
  5. 如果日志显示连接被旧电脑占用，先在旧电脑上停止 bridge：python3 plugins/codex-usage-stick/scripts/start_bridge.py --stop。
当前安全取舍

当前工作版本为了稳定显示 quota，BLE NUS 写入是开放的，没有启用 passkey 加密。它只发送本机 usage 百分比、token 数和状态字符串，不发送代码或外部网络数据。后续如果要把硬件审批也做到生产级，建议单独恢复加密配对或改为 USB Serial 专用审批链路。

什么时候选哪条链路

你的目标 选择 原因
日常看 Codex pet 和 quota BLE + USB 供电 最接近桌面摆件体验，线只供电，不占数据链路。
刚刷机、刚换 pet、调固件 USB-C 刷固件和上传 LittleFS 必须依赖 USB 数据连接。
蓝牙不好使，先证明屏幕没问题 USB Serial 串口直接喂 JSON，能快速判断问题在 BLE 还是屏幕解析。
把同一个 StickS3 换到另一台电脑 先停旧 bridge，再用新电脑 BLE 一个 StickS3 最好同一时间只被一台电脑的 bridge 占用。
想做完全脱离 Mac 蓝牙的局域网版本 未来可做 Wi-Fi 硬件支持 2.4GHz Wi-Fi，但当前仓库还没有实现 Wi-Fi 通信链路。

实现依据

  • src/ble_bridge.cpp：实现 Nordic UART Service，设备名广播为 Codex-XXXX，Mac bridge 写入 NUS RX。
  • src/data.h：同时读取 Serial 和 BLE buffer，只要收到 JSON 行就更新状态。
  • plugins/codex-usage-stick/scripts/start_bridge.py：Codex 插件启动和复用后台 BLE bridge。
  • plugins/codex-usage-stick/scripts/codex_usage_ble_bridge.py：读取本机 Codex usage，并通过 BLE 写 JSON 包。
  • tools/test_serial.py / tools/test_xfer.py：USB 串口调试和文件传输验证脚本。
  • M5Stack StickS3 官方文档：确认 StickS3 使用 ESP32-S3-PICO-1-N8R8，并列出 2.4GHz Wi-Fi、USB Type-C、250mAh 电池等硬件信息。
  • Espressif ESP32-S3-PICO-1 数据手册：确认该 SiP 支持 2.4GHz Wi-Fi 和 Bluetooth LE。
